# FlashFloppy on RP2350 (Raspberry Pi Pico 2)

This port runs the standard FlashFloppy *shugart* firmware on a
Raspberry Pi Pico 2 (RP2350), with disk images stored on an **SD card**
(SPI mode) instead of a USB stick. It remains bare-metal: no pico-sdk,
no TinyUSB.

## How it works

- `clk_sys` runs at **144MHz** so the PIO flux engine, at two PIO cycles
  per tick, reproduces the exact 72MHz SAMPLECLK used by the STM32
  firmware. All image decoding and bitcell arithmetic is unchanged.
- **RDATA** is generated by PIO0 SM0: DMA streams 16-bit flux intervals
  from the same ring buffer the STM32 code uses into the TX FIFO; the SM
  emits a fixed 388ns active-low pulse per interval.
- **WDATA** is captured by PIO0 SM1: a free-running 16-bit counter is
  sampled at each falling edge and pushed through DMA into the write
  ring, mimicking the STM32 timer input-capture semantics.
- Bus outputs are **emulated open-drain** (drive low / high-impedance via
  output-enable). Drive select/deselect asserts or releases the whole
  bus with a single SIO `GPIO_OE` write.
- The firmware is copied to **SRAM at boot** and runs entirely from RAM:
  flash (XIP) timing can never disturb flux generation. QSPI flash is
  touched only to load the image and to store `FF.CFG` settings (final
  4kB sector below the 1MB boundary, via bootrom flash routines).
- `verify PIO timing`: `python3 scripts/check_pio_timing.py`.

## Wiring notes

- RP2350 inputs are 5V-tolerant, so the floppy-bus inputs connect
  directly. Outputs are emulated open-drain and rely on the host's bus
  termination/pull-ups (all standard host interfaces have them). If you
  drive a bare drive-less setup, add ~1k pull-ups to 5V on the outputs.
- SD card: any 3.3V SPI micro-SD breakout. Runs at 400kHz during init,
  then 24MHz.
- Display: standard FlashFloppy I2C SSD1306/SH1106 OLED or HD44780+
  PCF8574 LCD on GPIO 20/21 (3.3V; external 2.2k-4.7k pull-ups
  recommended). A TM1651 3-digit LED display is also supported on the
  same two pins.
- JC (GPIO28): leave open for Shugart, strap to GND for IBM PC mode
  (same as the Gotek JC jumper).

## Building

```
make target mcu=rp2350 target=shugart level=prod   # one target
make prod-rp2350                                   # same thing
make debug-rp2350                                  # serial console build
```

Output: `out/rp2350/<level>/shugart/target.uf2`.

## Flashing

Hold BOOTSEL while plugging the Pico 2 into USB, then copy `target.uf2`
to the `RP2350` drive. There is no FlashFloppy bootloader on this
platform: firmware update is always via BOOTSEL.

## Limitations (v1 of the port)

- *shugart* target only: `apple2`, `quickdisk`, `io_test`, and the
  legacy bootloader/`bl_update` targets are not built for RP2350.
- USB stick storage is not supported (SD card only).
- FF OSD (I2C display OSD device) is not supported.
- The `CHGRST_pa14` (dedicated change-reset input) config option is not
  supported; use `chgrst=step` or a delay.
- Index-pulse alignment to the flux stream is accurate to roughly one
  flux sample (a few microseconds per revolution) rather than the STM32's
  sub-sample precision. This is well inside host tolerances.
